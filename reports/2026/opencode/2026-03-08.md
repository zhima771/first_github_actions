### fix(app): messages not loading reliably
**SHA**: `c797b60` | 🔗 [查看提交](https://github.com/anomalyco/opencode/commit/c797b60069df9b6510442e6e2d582c572f88d5c1)



**🎯 变更类型**：Bug 修复 + 重构

**⚡ 重要程度**：🔴高

**📋 变更摘要**：修复会话消息加载不可靠的问题，将历史窗口管理逻辑从页面组件中提取为独立模块。新增自动填充历史和滚动位置保持机制，显著改善长会话的消息加载体验。

**🎯 影响范围**：
- `packages/app/src/pages/session.tsx` - 会话页面主组件
- `packages/app/src/pages/session/history-window.ts` - 新增历史窗口管理模块
- `packages/app/src/pages/session/history-window.test.ts` - 新增单元测试

**🔍 技术洞察**：
- **架构影响**：将 200+ 行的历史窗口逻辑从页面组件解耦为独立可测试模块，遵循单一职责原则。新增 `historyLoadMode` 和 `historyRevealTop` 等纯函数，提升代码可维护性。
- **性能影响**：引入 `requestAnimationFrame` 调度历史填充，避免渲染阻塞。添加预取冷却机制（400ms）和连续无增长限制（2 次），防止过度请求。
- **安全考虑**：无直接安全影响，但改进的会话 ID 验证逻辑可防止跨会话数据泄漏。

**⚠️ 潜在风险**：
1. 滚动位置计算逻辑复杂，边界条件（如消息删除、网络失败）可能触发异常
2. `scheduleHistoryFill` 依赖多个响应式信号，信号竞争可能导致意外触发
3. 新增的自动加载可能在弱网环境下造成用户体验卡顿

**💡 关注建议**：
- **开发者**：关注 `historyRevealTop` 的滚动位置计算逻辑，建议增加集成测试覆盖滚动边界场景
- **用户**：长会话加载体验将显著改善，如遇滚动异常可清除本地缓存重试
- **审查重点**：验证 `preserve` 和 `reveal` 函数在消息动态增减时的滚动位置保持正确性

### fix: prune and evict stale app session caches (#16584)
**SHA**: `a139e92` | 🔗 [查看提交](https://github.com/anomalyco/opencode/commit/a139e9297d2a269308c66efbc7ed2b7a53a59a16)



**🎯 变更类型**：性能优化 + Bug 修复

**⚡ 重要程度**：🔴高

**📋 变更摘要**：实现了会话缓存的自动清理和驱逐机制，解决了长期运行应用中 stale session 数据累积导致的内存泄漏问题。通过 LRU 策略管理每个目录的会话缓存，确保只保留活跃会话的相关数据。

**🎯 影响范围**：
- `packages/app/src/context/global-sync/` - 全局状态同步核心模块
- `packages/app/src/context/sync.tsx` - 会话数据加载与缓存管理
- `packages/app/src/pages/layout.tsx` - 页面预取逻辑
- `packages/app/src/pages/session.tsx` - 会话页面导航处理

**🔍 技术洞察**：
- **架构影响**：引入了分层的缓存驱逐策略，在 `session-cache.ts` 中抽象了通用的缓存清理逻辑 (`dropSessionCaches`) 和驱逐选择逻辑 (`pickSessionCacheEvictions`)，实现了关注点分离。每个目录维护独立的 LRU 跟踪 (`seen` Map)，支持多工作区场景。
- **性能影响**：显著降低内存占用，防止 `message`、`part`、`session_diff`、`todo` 等状态对象无限增长。设置了 `SESSION_CACHE_LIMIT = 40` 和 `maxDirs = 30` 的双重限制，避免大规模项目下的性能退化。
- **安全考虑**：无直接安全影响，但正确的数据清理可防止敏感会话数据在内存中残留过久。

**⚠️ 潜在风险**：
1. **竞态条件**：`tracked()` 检查在异步加载完成后验证会话是否仍被跟踪，但存在加载过程中会话被驱逐的边缘情况，可能导致数据不一致。
2. **缓存过度驱逐**：LRU 策略可能过早驱逐用户即将访问的会话，尤其是当 `SESSION_CACHE_LIMIT` 设置较低时。
3. **状态同步复杂性**：新增的 `evict` 方法暴露给外部调用，需要确保调用方正确传递 `directory` 参数，否则可能导致跨目录状态污染。
4. **测试覆盖**：虽然添加了单元测试，但缺少对并发访问和边界条件（如快速切换会话）的集成测试。

**💡 关注建议**：
- **对开发者**：监控 `seen` Map 的大小增长，确保 `maxDirs` 限制在真实场景中足够；考虑添加缓存命中率指标以便调优 `SESSION_CACHE_LIMIT`。
- **对维护者**：建议添加调试日志记录驱逐事件，便于排查用户报告的"数据丢失"问题；考虑将缓存配置暴露为用户可选项。
- **代码审查重点**：验证 `tracked()` 检查在所有异步路径中的一致性；确保 `evict` 调用不会在渲染周期中触发状态更新。
