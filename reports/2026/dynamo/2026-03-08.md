#### 🔴 高重要度变更 (1)

### refactor: replace kube-rbac-proxy sidecar with controller-runtime WithAuthenticationAndAuthorization (#7045)
**SHA**: `7dbebf3` | 🔗 [查看提交](https://github.com/ai-dynamo/dynamo/commit/7dbebf3c61bd6adeb62d285f56f4d9434afb1f30)



**🎯 变更类型**：重构

**⚡ 重要程度**：🔴高

**📋 变更摘要**：移除 kube-rbac-proxy sidecar 容器，改用 controller-runtime 内置的 WithAuthenticationAndAuthorization 过滤器实现 metrics 端点的认证授权。简化部署架构的同时保持安全级别，默认启用安全服务并将 metrics 绑定地址从 127.0.0.1 改为 0.0.0.0。

**🎯 影响范围**：
- Helm Charts 部署配置 (operator deployment, RBAC, Service)
- Operator 主程序 (main.go 配置初始化)
- API 配置类型 (MetricsServer 结构体)
- 监控配置 (ServiceMonitor, metrics-service)
- 依赖项 (go.mod 新增 k8s.io/apiserver 等)

**🔍 技术洞察**：
- **架构影响**：部署模型从双容器 (manager + proxy) 简化为单容器，减少 Pod 复杂度和故障点；RBAC 权限从代理层转移到控制器运行时过滤器层
- **性能影响**：减少约 64Mi 内存和 5m CPU 的基础资源请求，消除 sidecar 与主容器间的网络跳转开销；但每次 metrics 请求需执行 SubjectAccessReview，增加 API Server 负载
- **安全考虑**：安全模型从外部代理认证转为内置过滤器认证，需要 TokenReview 和 SubjectAccessReview 集群级权限；默认启用 TLS (secure: true)，绑定地址变更需确保网络策略兼容

**⚠️ 潜在风险**：
1. **RBAC 权限升级**：新增的 ClusterRole 需要 authentication.k8s.io/tokenreviews 和 authorization.k8s.io/subjectaccessreviews 权限，在严格的多租户集群中可能需要额外审批
2. **API Server 压力**：每次 Prometheus scrape 都会触发 SubjectAccessReview，高频采集场景下可能增加 API Server 负载
3. **迁移兼容性**：现有部署升级时，旧的 Service/ServiceMonitor 配置可能与新配置不兼容，需要协调切换
4. **网络策略调整**：metrics 端口从 8443 (proxy) 变为 8080 (direct)，且绑定地址从 127.0.0.1 变为 0.0.0.0，现有 NetworkPolicy 可能需要更新

**💡 关注建议**：
- **给开发者**：升级后验证 RBAC 权限是否正确应用，测试 Prometheus 能否正常采集 metrics；监控 API Server 的 SubjectAccessReview 请求量变化
- **给运维人员**：升级前备份现有 deployment 配置，确认集群允许创建 ClusterRole/ClusterRoleBinding；检查 NetworkPolicy 是否允许 8080 端点的 HTTPS 流量
- **给安全团队**：审查新增的集群级 RBAC 权限，评估在多租户环境中的影响；确认 WithAuthenticationAndAuthorization 过滤器的安全等价性
- **升级策略**：建议采用蓝绿部署或金丝雀发布，先在小范围集群验证后再全面推广
