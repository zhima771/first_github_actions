#### 🔴 高重要度变更 (2)

### [Frontend] Add GPU-less render serving path (`vllm launch render`) (#36166)
**SHA**: `4497431` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/4497431df654e46fb1fb5e64bf8611e762ae5d87)



**🎯 变更类型**：架构变更
**⚡ 重要程度**：🔴高
**📋 变更摘要**：新增 GPU 无关的渲染服务路径（`vllm launch render`），将聊天/补全请求的预处理（tokenization、rendering）与 GPU 推理分离。删除旧的 `LaunchEngineClient` 实现，重构为独立的渲染服务模块，支持纯 CPU 部署渲染服务。
**🎯 影响范围**：
- vllm/entrypoints/cli/launch.py（CLI 入口）
- vllm/entrypoints/openai/api_server.py（API 服务器核心）
- vllm/entrypoints/serve/render/（新增渲染服务模块）
- vllm/entrypoints/openai/chat_completion/api_router.py（聊天接口）
- vllm/entrypoints/openai/completion/api_router.py（补全接口）
- vllm/entrypoints/serve/instrumentator/health.py（健康检查）

**🔍 技术洞察**：
- **架构影响**：实现了渲染服务与推理服务的彻底解耦。之前 `LaunchEngineClient` 是 `EngineClient` 协议的空对象实现，现在改为独立的 `OpenAIServingRender` 类，架构更清晰。支持独立部署渲染层，可实现预处理水平扩展。
- **性能影响**：CPU -only 渲染服务可独立扩容，减轻 GPU 推理节点的预处理负载。对于高并发场景，可将 tokenization/rendering 密集型操作卸载到廉价 CPU 实例，提升整体吞吐量。
- **安全考虑**：保留了 `trust_request_chat_template` 验证机制，防止恶意模板注入。错误响应处理与原有服务保持一致，安全边界未降低。

**⚠️ 潜在风险**：
1. **代码重复维护**：`OpenAIServingRender` 大量复制了 `OpenAIServingChat` 和 `OpenAIServingCompletion` 的逻辑（如 `_preprocess_chat`、`_preprocess_completion`），未来功能迭代需同步三处代码，维护成本高。
2. **健康检查语义变化**：渲染服务器始终返回 200 健康状态（无引擎可检查），可能导致监控系统无法检测渲染服务实际故障。
3. **功能不一致风险**：渲染服务不支持 LoRA、响应 API（ResponsesRequest）等功能，但错误提示可能不够明确，用户易混淆。
4. **删除文件影响**：`vllm/v1/engine/launch.py` 被完全删除，如有外部依赖此模块的集成将断裂。

**💡 关注建议**：
- **对开发者**：建议在 `OpenAIServingRender` 中通过继承或组合复用 `OpenAIServingChat/Completion` 的预处理逻辑，减少代码重复。添加集成测试验证渲染服务与完整服务的输出一致性。
- **对运维**：渲染服务健康检查需配置独立的监控指标（如请求延迟、错误率），不能仅依赖 `/health` 端点。建议文档明确说明渲染服务的能力边界（不支持生成、LoRA 等）。
- **对用户**：使用 `vllm launch render` 部署时，需配合推理服务使用。适用于需要大规模预处理但推理资源有限的场景，如多租户 SaaS 架构中的请求网关层。

### [Core] NGram GPU Implementation compatible with Async Scheduler (#29184)
**SHA**: `a6be75d` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/a6be75dbd2a8dd1886da725727ee178f42e3f84f)



**🎯 变更类型**：功能增强/性能优化
**⚡ 重要程度**：🔴高
**📋 变更摘要**：实现了基于 GPU 加速的 N-gram 投机采样提议器，并使其兼容异步调度器（Async Scheduler）。通过向量化 CUDA 操作替代原有的 CPU 查找逻辑，消除了 CPU-GPU 同步瓶颈，提升了无独立 Draft 模型场景下的推理吞吐量。
**🎯 影响范围**：核心推测解码模块 (`vllm/v1/spec_decode`)、调度器配置 (`vllm/config`)、Worker 执行路径 (`vllm/v1/worker`)、编译后端 (`vllm/compilation`)

**🔍 技术洞察**：
- **架构影响**：将 N-gram 匹配逻辑完全迁移至 GPU 端， aligns 了 v1 架构的全 GPU 设计理念。新增 `NgramProposerGPU` 类，通过 `unfold` 和 `gather` 操作实现并行后缀匹配，避免了传统 N-gram 在 CPU 上的序列查找开销。
- **性能影响**：显著减少了 Host-Device 数据传输和同步等待时间。支持异步调度意味着在模型计算期间可以并行进行 token 提议，进一步提升 GPU 利用率。但禁用了 torch.compile 缓存 (`disable_cache = True`)，可能导致首次运行编译时间增加。
- **安全考虑**：无直接安全漏洞引入。主要关注点在于张量操作中的索引边界检查（代码中已使用 `clamp` 处理），防止越界访问。

**⚠️ 潜在风险**：
- **异步同步风险**：引入了独立的 CUDA Stream 和 Event (`_num_valid_draft_tokens_copy_stream`) 用于异步 D2H 传输，若事件等待逻辑不当可能导致竞态条件或死锁。
- **显存开销**：新增了多个常驻显存的张量 buffer（如 `token_ids_gpu_tensor`, `num_tokens_no_spec_gpu`），在最大序列数较大时会增加显存占用。
- **编译兼容性**：显式禁用了 torch.compile 缓存，可能影响服务冷启动速度，且需验证后续 PyTorch 版本升级后是否仍需此限制。
- **逻辑复杂性**：`gpu_model_runner.py` 中的状态更新逻辑变得更加复杂，特别是针对 `ngram_gpu` 的特殊分支处理，增加了维护难度。

**💡 关注建议**：
- **开发者**：密切监控 `torch.compile` 缓存禁用对启动延迟的影响，探索后续启用缓存的可能性。加强异步流同步的单元测试，覆盖边界条件。
- **用户**：在启用 `ngram_gpu` + `async_scheduling` 时，建议先在小批量下验证准确率（Acceptance Rate），确保与 CPU 版本一致。关注显存占用变化，必要时调整 `max_num_seqs`。
- **审查重点**：检查 `update_scheduler_for_invalid_drafts` 中的同步逻辑，确保 CPU 端获取的 `num_valid_draft_tokens` 一定是 GPU 端计算完成后的最新值。

### [Model] Nano Nemotron VL - fast media preprocessing (#35657)
**SHA**: `b7332b0` | 🔗 [查看提交](https://github.com/vllm-project/vllm/commit/b7332b058c3b0d8533395b49dea9273aa0973b4e)



**🎯 变更类型**：性能优化/重构
**⚡ 重要程度**：🟡中
**📋 变更摘要**：将 Nano Nemotron VL 的媒体预处理从 PIL/Torchvision 迁移至纯 PyTorch 张量操作。移除 `torchvision.transforms` 依赖，利用 `torch.nn.functional.interpolate` 加速图像/视频缩放与归一化。
**🎯 影响范围**：`nano_nemotron_vl.py` 文件，涉及图像动态分块、视频帧处理及媒体处理器逻辑。
**💡 关注建议**：
1. **数值一致性**：PyTorch bicubic 与 PIL 实现存在细微差异，需回归测试验证推理精度是否受影响。
2. **内存开销**：早期将图片转为 Tensor 可能增加峰值内存，建议监控高分辨率输入下的显存/CPU 内存变化。
3. **依赖检查**：确认移除 `torchvision` 引用后，无隐式依赖冲突。
